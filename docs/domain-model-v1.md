# Domain Model v1

## Purpose

This document defines the core domain model for PriceAlert / DealHunt v1.

The goal is to create a domain model that is:

- clean and UI-neutral
- ready for TUI and future web/API use
- compatible with MariaDB as primary storage
- structured enough for growth without over-engineering the POC

We are using **Option B** as the foundation.

---

## Design Principles

### 1. Domain first
The domain model is the center of the system.

- TUI depends on domain data
- worker depends on domain data
- JSON contracts are derived from domain data
- database schema supports the domain, not the other way around

### 2. Separate noisy data from user-facing insights
Marketplace data is noisy, so we separate:

- raw scraped results
- grouped / deduplicated results
- aggregated snapshot

### 3. Keep models UI-neutral
Domain models must not contain TUI-only fields such as:

- selected
- focused
- expanded
- highlight_color

### 4. Snapshot is a first-class concept
MarketSnapshot is not just a temporary calculation.
It is a core object that represents the market state at a given time.

### 5. Events should be consistent
All important system events should follow one structure, regardless of whether they are shown in TUI, saved to DB, or sent to Telegram.

Core entities for v1:

## 1. TrackedKeyword

### Purpose
Represents a keyword configured by the user for monitoring.

### Responsibilities
- stores tracking configuration
- acts as the root entity for monitoring
- links scans, snapshots, price history, and alerts

### Fields

| Field | Type | Required | Notes |
|---|---|---:|---|
| id | string | yes | unique identifier |
| keyword | string | yes | the search phrase, e.g. `minyak goreng 2L` |
| basic_filter | string \| null | no | optional lightweight filter, e.g. `2L` |
| threshold_price | int64 \| null | no | threshold in smallest currency unit if needed |
| interval_minutes | int | yes | scan interval in minutes |
| telegram_enabled | bool | yes | whether Telegram alerts are active |
| status | enum | yes | `active`, `paused`, `archived` |
| created_at | datetime | yes | creation timestamp |
| updated_at | datetime | yes | update timestamp |

### Notes
- `keyword` should remain user-facing and unmodified.
- normalization for search processing should happen elsewhere, not overwrite `keyword`.
- `archived` is useful later even if v1 mostly uses `active` and `paused`.

---

## 2. ScanJob

### Purpose
Represents one execution of a monitoring cycle for a tracked keyword.

### Responsibilities
- records execution lifecycle
- links to raw/grouped results
- helps with debugging and observability

### Fields

| Field | Type | Required | Notes |
|---|---|---:|---|
| id | string | yes | unique identifier |
| tracked_keyword_id | string | yes | foreign key to TrackedKeyword |
| started_at | datetime | yes | scan start time |
| finished_at | datetime \| null | no | scan end time |
| status | enum | yes | `running`, `success`, `failed` |
| error_message | string \| null | no | populated when failed |
| raw_count | int | no | number of raw listings fetched |
| grouped_count | int | no | number of grouped listings produced |

### Notes
- `raw_count` and `grouped_count` can be set after processing completes.
- keeping ScanJob explicit will help later for worker diagnostics.

---

## 3. RawListing

### Purpose
Represents a single noisy listing result directly derived from scraped marketplace data.

### Responsibilities
- stores raw-ish marketplace output
- supports audit/debugging
- acts as input to grouping and deduplication

### Fields

| Field | Type | Required | Notes |
|---|---|---:|---|
| id | string | yes | unique identifier |
| scan_job_id | string | yes | foreign key to ScanJob |
| source | string | yes | e.g. `tokopedia` |
| title | string | yes | raw listing title |
| normalized_title | string | yes | normalized version used for grouping |
| seller_name | string | yes | seller display name |
| price | int64 | yes | observed current price |
| original_price | int64 \| null | no | crossed-out or reference price if present |
| is_promo | bool | yes | derived from marketplace data when possible |
| url | string | yes | listing URL |
| scraped_at | datetime | yes | fetch timestamp |

### Notes
- `normalized_title` should be deterministic and generated by domain/service logic.
- `original_price` may be absent and must not be assumed.
- raw listing does not try to represent “truth”, only what was observed.

---

## 4. GroupedListing

### Purpose
Represents a cleaned, grouped, user-facing listing candidate after deduplication.

### Responsibilities
- reduces noise
- powers top deals and market summary
- represents the best candidate for a product group

### Fields

| Field | Type | Required | Notes |
|---|---|---:|---|
| id | string | yes | unique identifier |
| scan_job_id | string | yes | foreign key to ScanJob |
| group_key | string | yes | deterministic grouping identity |
| representative_title | string | yes | human-readable title for the group |
| representative_seller | string | yes | selected representative seller |
| best_price | int64 | yes | lowest price in the group |
| original_price | int64 \| null | no | optional reference price |
| is_promo | bool | yes | whether representative listing is promo |
| listing_count | int | yes | number of raw listings in the group |
| sample_url | string | yes | URL of selected representative listing |

### Notes
- `group_key` should be stable enough for grouping logic, but not exposed as a UI concept.
- `listing_count` is important because it gives confidence that the price is not a one-off noise result.

---

## 5. MarketSnapshot

### Purpose
Represents aggregated market state for one tracked keyword at one moment in time.

### Responsibilities
- powers the TUI snapshot panel
- supports trend analysis
- acts as the main summary contract for future API/web

### Fields

| Field | Type | Required | Notes |
|---|---|---:|---|
| id | string | yes | unique identifier |
| tracked_keyword_id | string | yes | foreign key to TrackedKeyword |
| scan_job_id | string | yes | foreign key to ScanJob |
| min_price | int64 \| null | no | null when no usable data |
| avg_price | int64 \| null | no | null when no usable data |
| max_price | int64 \| null | no | null when no usable data |
| raw_count | int | yes | total raw listings |
| grouped_count | int | yes | total grouped listings |
| signal | enum | yes | `BUY_NOW`, `GOOD_DEAL`, `NORMAL`, `NO_DATA` |
| snapshot_at | datetime | yes | snapshot timestamp |

### Notes
- allowing price fields to be null makes `NO_DATA` cleaner and explicit.
- TUI should consume MarketSnapshot directly rather than recomputing it on screen.

---

## 6. PricePoint

### Purpose
Represents historical price information used for lightweight trend/history analysis.

### Responsibilities
- supports recent history panel
- supports alert logic such as new lowest or price drop
- preserves historical market summary at fixed points in time

### Fields

| Field | Type | Required | Notes |
|---|---|---:|---|
| id | string | yes | unique identifier |
| tracked_keyword_id | string | yes | foreign key to TrackedKeyword |
| scan_job_id | string | yes | foreign key to ScanJob |
| min_price | int64 \| null | no | historical min |
| avg_price | int64 \| null | no | historical avg |
| max_price | int64 \| null | no | historical max |
| recorded_at | datetime | yes | point-in-time timestamp |

### Notes
- in practice this can mirror snapshot history, but keeping it explicit makes history logic cleaner.
- if implementation simplicity is needed, v1 may persist it from MarketSnapshot values.

---

## 7. AlertRule

### Purpose
Represents a configurable rule that determines when an alert should fire.

### Responsibilities
- provides extensibility for alert logic
- separates alert configuration from alert execution

### Fields

| Field | Type | Required | Notes |
|---|---|---:|---|
| id | string | yes | unique identifier |
| tracked_keyword_id | string | yes | foreign key to TrackedKeyword |
| rule_type | enum | yes | see enum section |
| value | string | yes | rule parameter stored as string/serialized value |
| enabled | bool | yes | whether the rule is active |
| created_at | datetime | yes | creation timestamp |

### Notes
- for v1, this can be minimal even if not all rule types are exposed in UI.
- `value` can later evolve to JSON or structured sub-fields if necessary.

---

## 8. AlertEvent

### Purpose
Represents a system event or alert emitted during monitoring.

### Responsibilities
- powers activity log
- records alert behavior for audit/debugging
- supports external notification tracking

### Fields

| Field | Type | Required | Notes |
|---|---|---:|---|
| id | string | yes | unique identifier |
| tracked_keyword_id | string | yes | foreign key to TrackedKeyword |
| scan_job_id | string \| null | no | optional link to scan job |
| level | enum | yes | `INFO`, `ALERT`, `WARN`, `ERROR` |
| event_type | enum | yes | see enum section |
| message | string | yes | displayable event text |
| payload_json | string \| null | no | optional structured payload |
| sent_to_telegram | bool | yes | whether Telegram delivery was attempted/succeeded depending on chosen semantics |
| created_at | datetime | yes | event timestamp |

### Notes
- `message` should already be human-readable.
- `payload_json` is useful for debugging without polluting the main schema.

## Relationships

### Primary relationships

```text
TrackedKeyword
  ├── ScanJob
  │     ├── RawListing[]
  │     ├── GroupedListing[]
  │     └── MarketSnapshot
  │
  ├── PricePoint[]
  ├── AlertRule[]
  └── AlertEvent[]
```

### Relationship notes
- one TrackedKeyword has many ScanJobs
- one ScanJob has many RawListings
- one ScanJob has many GroupedListings
- one ScanJob has one MarketSnapshot
- one TrackedKeyword has many PricePoints
- one TrackedKeyword has many AlertRules
- one TrackedKeyword has many AlertEvents

## DTO / JSON Contract Guidance

The domain entities above should not always be returned directly as API or renderer output.

Recommended response shapes:
- DashboardState
- KeywordDetail
- GroupedListingView
- AlertEventView
- MarketSnapshotView

This allows the application to keep domain integrity while giving TUI and future web/API a clean contract.

### Example mapping idea
- TrackedKeyword -> keyword summary DTO
- MarketSnapshot -> snapshot DTO
- GroupedListing[] -> top deals DTO
- AlertEvent[] -> activity log DTO
- PricePoint[] -> recent history DTO

---

## Implementation Guidance

### What should live in domain/service logic
- title normalization
- grouping key generation
- snapshot calculation
- signal calculation
- alert rule evaluation
- event emission

### What should NOT live in domain entities
- SQL-specific tags as design drivers
- TUI formatting logic
- Telegram formatting logic
- screen selection state

---

## v1 Recommendation

For implementation, keep the model complete at the design level, but remain pragmatic in code.

Suggested practical approach:
- implement all eight entities conceptually
- persist the most important ones first
- allow AlertRule and PricePoint to start simple
- use MarketSnapshot and AlertEvent as first-class outputs early

This keeps the design future-ready without making the first milestone too heavy.
